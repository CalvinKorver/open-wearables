Railway provides the fastest way to get a production Open Wearables instance running. The template deploys all required services with sensible defaults - no manual infrastructure setup needed.

<Tip>
An admin account is created automatically on first startup (`admin@admin.com` / `your-secure-password`). To use custom credentials, set `ADMIN_EMAIL` and `ADMIN_PASSWORD` on the **Backend** service **before** the first successful deploy so the seed picks them up (as shown on the video).
</Tip>

<Warning>
**Changing `ADMIN_EMAIL` / `ADMIN_PASSWORD` after deploy does not reset the login.** The seed script `backend/scripts/init/seed_admin.py` only runs when **no** developer row exists. If a developer was already created, new env vars are ignored until you either:

1. **Sign in with the account that was actually seeded** тАФ usually the **original** defaults (`admin@admin.com` / `your-secure-password`) if you did not set custom vars before first boot, then change password while logged in via `POST /api/v1/auth/change-password` (Bearer token required) or the dashboard if available.

2. **Re-seed from env** тАФ Connect to Postgres (Railway **PostgreSQL** service тЖТ **Data** / CLI), delete existing admins, then redeploy **Backend**:
   ```sql
   DELETE FROM developer;
   ```
   On the next Backend start, migrations run and `seed_admin.py` creates one developer using **current** `ADMIN_EMAIL` and `ADMIN_PASSWORD`.

Variable names on the **Backend** service must match what the app expects: **`ADMIN_EMAIL`** and **`ADMIN_PASSWORD`** (wired in `backend/app/config.py` as `admin_email` / `admin_password`).
</Warning>

## What gets deployed

The Railway template provisions seven services:

| Service | Description |
|---------|-------------|
| **PostgreSQL** | Primary database for users, health data, and API keys |
| **Backend** | FastAPI application server |
| **Frontend** | React dashboard |
| **Redis** | Broker for Celery task queue and caching |
| **Celery Worker** | Processes background tasks (data syncs, webhooks) |
| **Celery Beat** | Schedules recurring sync jobs |
| **Flower** | Celery monitoring dashboard |

All inter-service networking and environment variables are configured automatically by the template.

## Post-deploy setup

<Steps>
  <Step title="Configure provider OAuth credentials">
    To enable wearable data syncing, add OAuth credentials for each provider you want to support on the **Backend** service.

    <Tip>
    You only need to configure providers you plan to use. See [Supported Providers](/providers/supported) for setup instructions for each one.
    </Tip>

    **Required for every OAuth provider (including Strava):**

    | Variable | Service | Example |
    |----------|---------|---------|
    | `API_BASE_URL` | **Backend** | `https://backend-production-xxxx.up.railway.app` (your public Backend URL, no trailing slash) |
    | `{PROVIDER}_CLIENT_ID` | **Backend** | From the provider developer portal (Strava: [settings/api](https://www.strava.com/settings/api)) |
    | `{PROVIDER}_CLIENT_SECRET` | **Backend** | From the same portal |

    For Strava specifically, also set **Authorization Callback Domain** in the Strava app to your Backend hostname only (e.g. `backend-production-xxxx.up.railway.app` or `api.yourdomain.com` тАФ not `localhost`, and not the full callback path).

    OAuth redirect URIs are built automatically as:

    `{API_BASE_URL}/api/v1/oauth/{provider}/callback`

    If `API_BASE_URL` is unset, it defaults to `http://localhost:8000` and Strava (and other providers) will fail in production. If `{PROVIDER}_CLIENT_ID` is unset, the authorize URL is sent with an empty `client_id` and Strava rejects the request.

    After changing variables, **redeploy the Backend** (and Celery Worker if it handles OAuth-related tasks).

    See [Strava API Integration](/providers/strava-api-integration) for full Strava setup including webhooks.
  </Step>

  <Step title="Set up a custom domain (optional)">
    Railway assigns a default `*.up.railway.app` domain to each service. To use your own domain:

    1. Open your **Frontend** service in Railway
    2. Go to **Settings > Networking > Custom Domain**
    3. Add your domain and configure the DNS records as shown

    Repeat for the **Backend** service if you want a custom API domain. Then update the following environment variables:

    | Variable | Service | Value |
    |----------|---------|-------|
    | `VITE_API_URL` | Frontend | Your backend URL (e.g. `https://api.yourdomain.com`) |
    | `CORS_ORIGINS` | Backend | Your frontend URL (e.g. `https://yourdomain.com`) |
  </Step>
</Steps>

## Environment variables reference

Key variables you may want to customize on the **Backend** service:

| Variable | Description | Default |
|----------|-------------|---------|
| `API_BASE_URL` | Public Backend URL used for OAuth callbacks and webhooks | `http://localhost:8000` |
| `SECRET_KEY` | JWT signing key | Auto-generated by template |
| `CORS_ORIGINS` | Allowed CORS origins (comma-separated) | Frontend Railway URL |
| `FRONTEND_URL` | Dashboard URL (emails, redirects) | Frontend Railway URL |
| `STRAVA_CLIENT_ID` | Strava OAuth client ID | unset |
| `STRAVA_CLIENT_SECRET` | Strava OAuth client secret | unset |
| `LOG_LEVEL` | Application log level | `info` |
| `REDIS_SSL` | Enable TLS for Redis connections (`rediss://`). Required for managed Redis with transit encryption (e.g. AWS ElastiCache, Upstash with TLS). | `false` |

<Note>
Database and Redis connection strings are injected automatically by Railway тАФ you do not need to set these manually. `REDIS_SSL` defaults to `false` and is only needed if you swap Railway's built-in Redis for an external managed instance with TLS enabled.
</Note>
